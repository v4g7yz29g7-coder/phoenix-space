# 🐦‍🔥 ФЕНИКС (PHOENIX SPACE) — ТЕХНИЧЕСКИЙ МАНИФЕСТ

**Версия 1.0 — Июль 2026**

---

## 1. АБСТРАКТ

«Феникс» — это некоммерческий, открытый протокол для построения децентрализованной экосистемы (Цифрового Ковчега). Протокол объединяет суверенную идентичность (DID), прямые P2P-коммуникации (WebRTC) и децентрализованное хранилище артефактов (IPFS/блокчейн). Цель — предоставить людям инструменты для цифровой автономии, неподвластные цензуре и слежке.

---

## 2. АРХИТЕКТУРА

Система состоит из трёх слоёв:

### 2.1. Клиентский слой (Frontend / PWA)
- **Технологии:** HTML5, CSS3, Vanilla JavaScript (ES Modules).
- **Интерфейс:** Адаптивный PWA, работающий офлайн. Поддерживает 7 языков.
- **DID-менеджер:** Генерация мнемонической фразы (BIP39, 12 слов), создание пары ключей Ed25519. Приватный ключ хранится исключительно на устройстве пользователя (localStorage).
- **P2P-клиент:** Использует библиотеку `simple-peer` (WebRTC) для аудио/видео звонков и передачи файлов без центрального сервера.

### 2.2. Серверный слой (Координация)
- **Технологии:** Node.js, Express, Socket.io, SQLite (для временных данных).
- **Роль:** Исключительно сигналинг для установки P2P-соединений (обмен SDP и ICE-кандидатами).
- **Важно:** Сервер не хранит сообщения, файлы или историю звонков. Вся пользовательская коммуникация шифруется (DTLS-SRTP) и передаётся напрямую между пирами.

### 2.3. Слой данных и блокчейна
- **Локальное хранилище:** Все пользовательские данные (профиль, цели, артефакты) хранятся на клиенте.
- **Блокчейн (в разработке):** Интеграция с Polygon (Mumbai testnet) для регистрации NFT-артефактов. Смарт-контракт `ArtifactRegistry.sol` уже развёрнут локально.
- **IPFS (план):** Децентрализованное хранилище для файлов артефактов.

## 3. ИДЕНТИЧНОСТЬ (DID)

### 3.1. Генерация
1. Пользователь нажимает «Создать DID».
2. Клиент генерирует 12-словную мнемонику (энтропия 128 бит) с помощью `ethers.Wallet.createRandom()`.
3. Из мнемоники извлекается приватный ключ и адрес Ethereum.
4. DID формируется как `did:phoenix:0x<адрес>`.

### 3.2. Аутентификация
- Пользователь вводит свою мнемонику в поле входа.
- Клиент восстанавливает кошелёк и подписывает сообщение «Войти в Круг».
- Сервер проверяет подпись (`ethers.verifyMessage`) и, в случае успеха, авторизует соединение.
- Приватный ключ никогда не покидает устройство пользователя.

### 3.3. Репутация (Proof-of-Benefit, план)
- Каждое действие (создание артефакта, завершение практики) увеличивает счётчик `practices_completed` или `artifacts_count` в таблице `did_reputation`.
- При достижении пороговых значений статус повышается (новичок → практикующий → архитектор).
- В будущем репутация будет привязана к смарт-контракту на Polygon.

## 4. КОММУНИКАЦИИ (КРУГ СОРАТНИКОВ)

### 4.1. Текстовый чат
- Работает через WebSocket (Socket.io).
- Сообщения передаются в реальном времени в рамках «комнат».
- Поддерживается опциональная подпись сообщений приватным ключом DID для верификации отправителя.

### 4.2. Видео/аудио звонки (P2P)
- Используется WebRTC через библиотеку `simple-peer`.
- Сигнальный сервер (Socket.io) используется только для обмена SDP-предложениями/ответами и ICE-кандидатами.
- Медиа-потоки передаются напрямую между браузерами, минуя сервер.
- Для обхода NAT используется встроенный в браузер STUN. В будущем планируется добавление TURN-сервера для сложных сетевых конфигураций.

### 4.3. Передача файлов
- Файлы передаются через DataChannel внутри установленного WebRTC-соединения.
- Поддерживается передача любых типов файлов, размер ограничен только памятью устройства.

## 5. АРТЕФАКТЫ И БЛОКЧЕЙН

### 5.1. Создание артефакта
- Пользователь заполняет форму в «Мастерской» (название, тип, описание, файл).
- Артефакт сохраняется в локальной базе SQLite на сервере (до перехода на IPFS).
- При создании начисляется опыт ремесленника (+10 к репутации).

### 5.2. Запись в блокчейн
- Для каждого артефакта вычисляется `keccak256`-хеш контента.
- Вызывается смарт-контракт `ArtifactRegistry` (развёрнут на локальном Ganache, в планах — Polygon Mumbai).
- Контракт сохраняет ID артефакта, название, автора и хеш.
- Транзакция подтверждает авторство и временную метку.

---

## 6. БЕЗОПАСНОСТЬ

- **Приватные ключи:** Хранятся только на клиенте. Сервер не имеет к ним доступа.
- **Коммуникации:** Все P2P-соединения шифруются (DTLS-SRTP). Текстовые сообщения могут быть подписаны DID.
- **Сервер:** Не хранит пользовательский контент, выполняет роль координатора.
- **Защита от инъекций:** Все пользовательские входы валидируются (Zod) и экранируются.
